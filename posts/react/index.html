<p>React is, as the <a href="https://reactjs.org/" rel="noopener">website</a> says, &ldquo;a JavaScript library for building user interfaces.&rdquo; What this means exactly, is often confusing to people.</p> <p>Many articles claim to teach you &ldquo;React in 10 minutes&rdquo; but rarely go beyond a few code examples. The problem is that this leaves you with an inkling of what React looks like and maybe how to write a bit of it, but no clue <em>why</em> it works and how to think about fixing React bugs.</p> <p>This isnâ€™t some random tutorial which throws you a bunch of example code that you go on to copy-and-paste and then leaves you hanging wondering what it does. The goal of the workshop is to give attendees a good foundation of the concepts within React, why people choose to use React, and how it works behind the hood. Hopefully, this gives you a good enough foundation to hit the ground running and avoid entire classes of bugs when starting out in React!</p> <p>This post assumes that you know basic HTML, CSS, and JavaScript.</p> <h2 id="what-is-react">What is React?</h2> <p>The TLDR; is that it&rsquo;s a bunch of code you can add to your project that makes building websites and interfaces that <strong>react</strong> to changes when you do things (e.g. press buttons) really easy.</p> <p>React has two distinguishing features that set it apart:</p> <ol> <li>React is <strong>declarative</strong> meaning you tell React exactly what you want the page to look like and it&rsquo;ll figure out how to do that.</li> <li>React is <strong>component based</strong> meaning that you can create pieces of the UI that you can reuse across different parts of your application.</li> </ol> <h2 id="how-a-website-works">How a website works</h2> <p>Most likely, you&rsquo;ve written some code for a website in plain old HTML before. This, as you know, defines the structure for what is inside a webpage. If the webpage is a house, the HTML is the blueprint that tells you what the house should generally look like and what is inside.</p> <p>Yet, to turn that bunch of HTML into something displayable in your web browser, it needs to parse it into a more intermediate format it can more easily work with: the Document Object Model (DOM).</p> <p>The nested structure of the HTML lends itself very well to a tree-like structure which the browser can then efficiently traverse and make updates to.</p> <p> <img src="/posts/images/react/html-dom-site.png" alt="Turning HTML into the DOM and then into the actual site" /><em>Turning HTML into the DOM and then into the actual site</em></p> <h2 id="how-react-works">How React works</h2> <h3 id="the-virtual-dom">The Virtual DOM</h3> <p>Of course, you can still manipulate and interact with the regular DOM <a href="https://developer.mozilla.org/en-US/docs/Web/API/Document_Object_Model/Introduction" rel="noopener">through JavaScript</a> still but the main gripe that React has with this way of manipulating the DOM (e.g. adding new things to the screen or modifying existing bits on the page) is that it can be <em>realllyyy</em> slow in some cases.</p> <p>This is where the mystical <strong>virtual DOM</strong> comes in. Modifying the DOM in a declarative way is difficult because modifying the real DOM is imperative by nature, making it difficult to write declarative applications. You could use <code>element.innerHTML = 'your html'</code> in a declarative fashion, but an in-memory version implemented in JavaScript would be faster than using <code>innerHTML</code>, which is exactly exactly what React does. React creates a virtual DOM where it keeps track of what it would <em>like</em> the real DOM to look like and then intelligently batches and combines changes together to optimize and squeeze the most performance out of the few expensive writes to the real DOM that it actually does. More info on <a href="https://buildwithreact.com/tutorial/under-the-hood" rel="noopener">how DOM diffing and patching works</a> under the hood for those curious.</p> <blockquote> <p>Please note that Virtual DOM is not faster than raw, imperative operations on the real DOM (best performance). React uses the Virtual DOM because it is a relatively efficient way of declaratively representing your UI from state, and much faster compared to straight <code>innerHTML</code> calls.</p> </blockquote> <p> <img src="/posts/images/react/virtual-DOM.png" alt="React&amp;rsquo;s Virtual DOM in action. ReactDOM.render() tells React to attach your React components to the real DOM" /><em>React&rsquo;s Virtual DOM in action. ReactDOM.render() tells React to attach your React components to the real DOM</em></p> <h3 id="components">Components</h3> <p>Another things that React tackles really well is that regular HTML and JS makes it a lot more difficult than it should be to re-use the same UI and logic across different pages on your website.</p> <p> <img src="/posts/images/react/component-reuse.png" alt="What if you wanted to reuse this &amp;lsquo;product card&amp;rsquo; component?" /><em>What if you wanted to reuse this &lsquo;product card&rsquo; component?</em></p> <p>The main philosophy of functional React is that you shouldn&rsquo;t override and inherit behaviour. Instead, get the behaviour you want by creating and composing reusable <em>components</em>. Each of these components can hold some sort of data of its own, called its <strong>state</strong>. Each component can also take in a few arguments or parameters called <strong>properties</strong> or <strong>props</strong> for short. Each component is composed up of either primative HTML elements (e.g. <code>&lt;div&gt;</code> or <code>&lt;h1&gt;</code>) or other React components. The elements that a component is made up of are called its <strong>children</strong>.</p> <p> <img src="/posts/images/react/component-diagram.png" alt="React Component Diagram" /><em>React Component Diagram</em></p> <p>Because React uses composition to build components, there is a natural <em>downward flow of information</em>, where components pass data to their children. Thus, to change a parent&rsquo;s state, the parent needs to explicitly pass a callback function that allows that behaviour to the child (e.g. an arrow function that wraps the <code>setState</code> function for the parent).
